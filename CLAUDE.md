## Commands

```bash
npm run dev      # start dev server with HMR at http://localhost:4321
npm run build    # static build to ./dist
npm run preview  # serve the built ./dist locally
npm run astro -- check   # type-check .astro files (uses Astro's strict tsconfig)
```

There is no test suite, linter, or formatter configured. Type-checking rides on `astro/tsconfigs/strict`.

## Architecture

A single-page Astro 5 portfolio, statically generated. No UI framework — plain `.astro` components plus one client-side JS motion controller. Two moving parts matter most:

**1. Content is fully data-driven from one file: [src/data/site.js](src/data/site.js).**
Single source of truth for *all* content — profile, work categories, projects, about, CV, socials. Components read from it; don't hardcode content in a component. Key mechanics:
- `projects[]` drives three surfaces at once: a row on the home page, a card on `/gallery`, and a full case-study page at `/work/<id>`. Detail pages are generated by `getStaticPaths()` in [src/pages/work/[id].astro](src/pages/work/[id].astro), which also wires each project's "next project" link (wraps around at the end). A project's `id` is its URL slug and must be unique.
- A project's `cat` must match a `key` in `categories[]` — that linkage powers the Work section's filter tabs. Each surface prepends its own lead tab: the home page's [Work.astro](src/components/Work.astro) adds **"Recent"** (shows only the first 3 projects in `site.js` order); the gallery's [WorkGallery.astro](src/components/WorkGallery.astro) adds **"All"** (shows everything). Both share one filter contract in `main.js` (`data-filter` / `data-work-row` / `data-cat`), which special-cases `recent` and `all` and otherwise matches on `data-cat`.
- `/gallery` ([src/pages/gallery.astro](src/pages/gallery.astro)) is a **complete alternate homepage** — same sections as `/`, but the Work list is swapped for a card grid (`WorkGallery`). To make it the primary homepage, swap its body with [src/pages/index.astro](src/pages/index.astro).
- Images live in `/public`, referenced by absolute path (e.g. `/img/case-studies/foo.jpg`); external URLs also work. Missing `media`/`gallery` falls back to CSS placeholder tiles.
- The file is heavily commented as an editing guide — adding a case study is copy-paste-a-block, no component edits.

**2. All motion is one client-side module: [src/scripts/main.js](src/scripts/main.js).**
Imported once at the bottom of [src/layouts/Layout.astro](src/layouts/Layout.astro). Uses GSAP + ScrollTrigger + Lenis (smooth scroll). The build has no Astro integrations on purpose — `astro.config.mjs` is deliberately minimal. Conventions to preserve:
- Components expose behavior to `main.js` via `data-*` attributes (`data-reveal`, `data-split`, `data-work-row`, `data-marquee`, `data-cursor`, `.magnetic`, `data-filter`, `data-theme-toggle`, etc.). To animate new markup, add the matching `data-` hook rather than new selectors in JS.
- Everything is feature-detected and **must respect `prefers-reduced-motion`** and touch devices — follow the existing `reduceMotion`/`isTouch` guards (each `init*` function bails or renders a static end-state).
- `runPreloader()` is the boot sequence: preloader plays first, then hands off to `revealHero()` and initializes scroll-triggered effects. Start new scroll animations from inside its `start()` so `ScrollTrigger.refresh()` runs after they register.

**Theme** (dark/light) is resolved by an inline script in `<head>` of `Layout.astro` *before paint* (reads `localStorage.theme`, falls back to `prefers-color-scheme`) to avoid a flash; the runtime toggle in `main.js` just flips `document.documentElement.dataset.theme` and persists it. The `js` class is added to `<html>` early so no-JS fallback content only hides when JS actually runs.

**Styling** is global CSS in [src/styles/global.css](src/styles/global.css) plus scoped `<style>` blocks in components. Reuse the CSS custom properties instead of hardcoding: fluid type scale (`--step--1`, `--step-0`, `--step-2`, `--step-huge`), spacing (`--pad`), fonts (`--font-display`), theme colors (`--fg`, `--bg`, `--line`), and the per-project accent (`--a`).

**Before deploying:** set the real origin in `astro.config.mjs` (`site:`, currently `https://example.com`) — it feeds canonical/OG meta.

## Working guidelines

These bias toward caution over speed. For trivial tasks, use judgment.

1. **Think before coding.** State assumptions explicitly; if uncertain, ask. If multiple interpretations exist, present them — don't pick silently. If a simpler approach exists, say so.
2. **Simplicity first.** Minimum code that solves the problem, nothing speculative. No unrequested features, abstractions, flexibility, or error handling for impossible scenarios.
3. **Surgical changes.** Touch only what you must. Don't refactor what isn't broken; match existing style. Remove orphans *your* changes created; mention (don't delete) pre-existing dead code. Every changed line should trace to the request.
4. **Goal-driven execution.** Turn tasks into verifiable goals and, for multi-step work, state a brief plan with a verify step for each.
